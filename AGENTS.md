# Slide Agent Guide

This is the internal engineering and product reference for Slide. Keep
`README.md` customer-facing: what Slide is, how to run it, and where the main
pieces live. Put product choices, technical tradeoffs, release caveats, API
contracts, and agent-specific runbooks here.

## Product Direction

Slide is a phone-number-only video caller for people who already know each
other. The product bet is speed and absence of ceremony: no usernames, no
meeting links, no heavy visual chrome.

Current v1 scope:

- Native iOS and Android clients.
- Phone OTP auth.
- Contacts, recents, 1:1 and group calls.
- Screen share.
- WebRTC media through Slide-owned infrastructure.

Non-goals for v1:

- Public social graph.
- Meeting scheduling.
- Account handles.
- Rich profile surfaces.
- Marketing-site complexity beyond a clear landing page.

## Technical Shape

- iOS: SwiftUI, XcodeGen project in `ios/project.yml`, bundle id
  `app.exla.slide`.
- Android: Kotlin/Compose, Gradle Kotlin DSL, Play application id `app.slide`.
  The Kotlin namespace remains `ai.exla.slide`; do not confuse namespace with
  the Play package name.
- Web: Next.js marketing site in `web/`.
- Backend: Rust `axum` API, `sqlx`, Postgres, Redis.
- Media: `webrtc-rs` SFU plus coturn for STUN/TURN.
- Shared core: `crates/slide-core` for models, JWT, OTP, E.164, and TURN creds.

## Design System

The identity is quiet and precise: white space, thin near-black type, hairlines,
and no decorative noise. Keep the system consistent across iOS, Android, and web.

Color tokens:

- `bg`: `#FFFFFF`, every background.
- `bgGrouped`: `#FAFAFA`, grouped sections.
- `text`: `#0A0A0A`, primary near-black text.
- `textSecondary`: `#6B7280`, secondary gray.
- `hairline`: `#ECECEC`, 1px borders and dividers.
- `accent`: `#0A0A0A`, primary action and active toggles.
- `danger`: `#E5484D`, end call, decline, log out. This is the only red.

Typography:

- System font: SF Pro on iOS, Roboto on Android, `-apple-system`/Inter on web.
- Weights: Light 300 for large headings, Regular 400 for body, Medium 500 only
  for buttons and active states.
- Small uppercase labels may use tracking around `0.02em`.
- The `Slide` wordmark is thin, around 300 weight, with tracking around `0.04em`.

Shape, spacing, and motion:

- 8px spacing grid. Touch targets at least 44px.
- Corner radius 12-16px, subtle.
- Prefer 1px hairlines over cards with shadows.
- Motion should be fast and subtle, around 150-200ms ease-out.
- Incoming call pulse is gentle, scale 1.0 to 1.04.
- Icons are thin line icons, about 1.5px stroke.
- Bottom navigation has three tabs: Calls, Contacts, Profile.
- Active calls take over full screen above tabs.

## API Contract

Base URL is `/v1`. All JSON uses camelCase. Auth uses
`Authorization: Bearer <accessToken>` except auth endpoints. Errors use:

```json
{ "error": { "code": "rate_limited", "message": "too many requests", "retryAfter": 30 } }
```

Auth:

| Method | Path | Body | Returns |
|---|---|---|---|
| POST | `/auth/request-otp` | `{ "phone": "+14155550123" }` | `{ "status": "sent" }`; with `EXPOSE_DEV_OTP=true` in console-only dev, also `{ "devCode": "123456" }` |
| POST | `/auth/verify-otp` | `{ "phone", "code" }` | `{ "accessToken", "refreshToken", "isNewUser", "user" }` |
| POST | `/auth/refresh` | `{ "refreshToken" }` | `{ "accessToken", "refreshToken" }` |
| POST | `/auth/logout` | `{ "refreshToken" }` | `204` |

User and onboarding:

| Method | Path | Body | Returns |
|---|---|---|---|
| GET | `/me` | none | `User` |
| PATCH | `/me` | `{ "displayName"?, "avatarUrl"? }` | `User` |
| POST | `/me/avatar` | multipart `file` | `{ "avatarUrl" }` |
| POST | `/devices` | `{ "pushToken", "platform", "appVersion" }` | `Device` |

Contacts:

| Method | Path | Body | Returns |
|---|---|---|---|
| POST | `/contacts/sync` | `{ "phones": ["+1..."], "names"?: ["Ada Lovelace"] }` | `[{ "phone", "displayName", "userId"?, "onSlide" }]` |
| GET | `/contacts` | none | `[Contact]` |

`names`, when present, is a parallel array aligned by index with `phones`.
Older clients may omit it; the server should preserve existing names instead of
replacing them with blanks.

For client rendering, treat `contactUserId` on `GET /contacts` and `userId` on
`POST /contacts/sync` as the source of truth for whether somebody is on Slide.
The `onSlide` boolean is compatibility sugar for sync results and should not be
required on stored contact rows.

Calls:

| Method | Path | Body | Returns |
|---|---|---|---|
| POST | `/calls` | `{ "type": "one_to_one"|"group", "participantUserIds": [], "videoEnabled"?: true, "ringStyle"?: "call"|"knock" }` | `{ "call", "joinToken", "sfuUrl", "iceServers" }` |
| POST | `/calls/:id/accept` | none | `{ "call", "joinToken", "sfuUrl", "iceServers" }` |
| POST | `/calls/:id/decline` | none | `204` |
| POST | `/calls/:id/leave` | none | `204` |
| GET | `/calls?cursor=` | none | `{ "calls": [Call], "nextCursor"? }` |

Realtime plane A, app signaling: `GET /v1/ws?token=<accessToken>`.

Server to client events: `incoming_call`, `call_accepted`, `call_declined`,
`call_ended`, `participant_joined`, `participant_left`, `presence_update`.
`incoming_call` includes `callId`, `callType`, `fromUserId`, `fromName`,
`videoEnabled`, `ringStyle`, `knock`, and `call`.

Client to server events: `presence_ping`, `heartbeat`, `knock`.

`ringStyle` defaults to `call`. `ringStyle: "knock"` is still a real call
invitation: it creates a `calls` row, sends an `incoming_call` event/push with
`knock: true`, and is answered through `/calls/:id/accept`. The WebSocket
`knock` event is only the live tap rhythm for already-open apps; it must not be
used as an offline push or CallKit/CallStyle substitute because it has no call
id or join token.

Models:

```text
User          { id, phone, displayName?, avatarUrl?, createdAt, lastSeenAt }
Device        { id, userId, pushToken, platform, appVersion, updatedAt }
Contact       { id, ownerUserId, contactUserId?, phone, displayName, avatarUrl? }
Call          { id, roomId, sfuNodeId, type, createdBy, status, videoEnabled,
                ringStyle,
                startedAt?, endedAt?, createdAt,
                participants: [{ userId, state, joinedAt?, leftAt? }] }
IceServer     { urls: [], username?, credential? }
```

Enums:

- `platform`: `ios`, `android`.
- `type`: `one_to_one`, `group`.
- `call.status`: `ringing`, `active`, `ended`, `missed`, `declined`.
- `participant.state`: `invited`, `ringing`, `joined`, `left`, `declined`.

## SFU Media Protocol

The SFU media socket is reached at:

```text
wss://<sfu-node>/ws?room=<roomId>&token=<joinToken>
```

The `joinToken` is a short-lived JWT minted by the control plane under the
shared `SFU_JWT_SECRET`. The SFU validates it and verifies that its `room_id`
matches the query before upgrading.

Client to SFU messages:

- `offer { sdp }`: client publishes tracks and asks to subscribe. SFU replies
  `answer`.
- `answer { sdp }`: reply to an SFU-initiated renegotiation `offer`.
- `ice { candidate, sdpMid?, sdpMlineIndex? }`: trickled ICE.
- `ping`.

SFU to client messages:

- `answer { sdp }`: answer to the client's offer.
- `offer { sdp }`: SFU-initiated offer when new remote tracks were added.
- `ice { candidate, sdpMid, sdpMlineIndex }`.
- `peer_joined { userId }`, `peer_left { userId }`.
- `pong`, `error { message }`.

Flow:

1. Client gets `joinToken`, `sfuUrl`, and `iceServers` from `/calls` or
   `/calls/:id/accept`.
2. Client builds an `RTCPeerConnection`, adds mic/cam tracks, creates an offer,
   and sends `offer`.
3. SFU sets remote, attaches forwarding tracks for media already in the room,
   and answers.
4. When another participant publishes, the SFU adds that forwarding track to
   each existing peer and sends an `offer`; clients reply `answer`.
5. RTP from each publisher uplink is pumped into per-subscriber local tracks.
   Simulcast layer selection slots into this stage later.

## Deployment Status

Current known live surfaces:

- Landing site: live at `https://slide.viraat.dev`, Vercel project `web`,
  including `/privacy`, `/terms`, and `/web`.
- Backend API: live on Fly at `https://slide-api.fly.dev`, Fly app
  `slide-api`, org `exla`.
- API health: `GET /v1/health` returns `ok`, verified June 6, 2026.
- Backend deploy command: `fly deploy -c deploy/fly/slide-api.fly.toml`.
- Backend image: built from `deploy/aws/Dockerfile.allinone` on each Fly deploy.
- Backend caveat: Fly runs a single API machine because app signaling is
  in-memory. Keep `min_machines_running = 1` and do not scale horizontally until
  the call signaling hub is backed by durable/shared state.
- iOS: signed IPA for `app.exla.slide`, v1.0.0, was uploaded to App Store
  Connect on May 30, 2026. App record: `Slide Video Calls`, Apple ID
  `1780017294`.
- Android: debug APK builds; Play release is gated on Play Console setup.

Production backend guidance:

- `DATABASE_URL` is a Fly secret pointing at managed Postgres. Redis is bundled
  in the all-in-one container unless a `REDIS_URL` secret is set.
- `sqlx` uses prepared statements, so use a direct Postgres connection or a
  session pooler on port 5432. Do not use a transaction pooler on 6543.
- `SMS_PROVIDER=console` is dev/demo only. Production needs Twilio or another
  real SMS provider.
- AWS App Runner is legacy for this app. Its ingress rejected WebSocket upgrades,
  which breaks the `/v1/ws` incoming-call signaling path. Use Fly for the API.
- The SFU media plane is separate from the API. Set `SFU_PUBLIC_URL` to the
  deployed SFU WebSocket endpoint and keep TURN settings current.

## Secret Handling

Never commit:

- `.env` or `.env.*` except `.env.example`.
- App Store `.p8`, `.pem`, `.p12`, or `ios/fastlane/.asc.env`.
- Android keystores, `android/keystore.properties`, or Play service-account JSON.
- Real phone numbers, videos, OTPs, live API tokens, or production credentials in
  docs, screenshots, tests, or issue comments.

Use obviously fake examples such as `+1 415 555 0137` and `a@example.com`.

Known security note: an AWS access key was previously pasted into chat. Treat it
as compromised and rotate it before relying on that account for production.
